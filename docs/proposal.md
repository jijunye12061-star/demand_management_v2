# proposal.md — 系统提案

---

## 1. 项目背景

现有「需求管理系统」基于 Streamlit 构建，已稳定运行 4 个月，支撑销售→研究员→管理员的需求流转。  
随着功能增长，Streamlit 在交互体验、移动端适配、并发性能方面已接近瓶颈。  
本次重构目标：**以前后端分离架构重写，100% 覆盖现有功能，同时解决已知技术债务**。

---

## 2. 核心功能边界

### 2.1 用户与认证

| 功能   | 说明                                 | 角色    |
|------|------------------------------------|-------|
| 登录   | JWT 认证, 替代 Streamlit session_state | 全部    |
| 修改密码 | 用户自行修改                             | 全部    |
| 用户管理 | 创建/删除/重置密码/分配角色与团队                 | admin |

**角色模型 (不变)**:

- `sales` — 销售: 提交需求, 查看进度
- `researcher` — 研究员: 处理任务, 上传结果, 也可代提需求
- `admin` — 管理员: 全局可见, 统计分析, 系统管理

### 2.2 需求流转 (核心工作流)

```
销售提交需求 ──────→ 待处理 (pending)
                        │
         ┌──────────────┼──────────────┐
         ↓              ↓              ↓
    研究员接受      研究员撤回      管理员重新分配
    → 处理中        → 归还销售      → 更换研究员
   (in_progress)    (pending)
         │
         ↓
    上传附件 + 填写说明 + 记录工时
         │
         ↓
    标记完成 (completed)
```

**需求提交字段**:

| 字段              | 必填 | 说明                                 |
|-----------------|----|------------------------------------|
| title           | ✅  | 事项名称                               |
| description     | —  | 内容描述                               |
| request_type    | ✅  | 枚举: 基金筛选/传统报告定制/量化策略定制/系统定制/综合暂时兜底 |
| research_scope  | —  | 枚举: 纯债/固收+/权益/量化/资产配置/其他           |
| org_name        | ✅  | 从用户所属团队的机构列表中选择                    |
| org_type        | 自动 | 随机构带入                              |
| department      | 条件 | 银行/券商/保险时必选子部门                     |
| researcher_id   | ✅  | 指定研究员                              |
| is_confidential | —  | 默认公开                               |
| created_at      | —  | 默认当前时间, 支持回溯选择                     |

**研究员提交需求**: 额外需要选择 `sales_id` (代哪个销售提)

### 2.3 保密机制

| 角色         | 可见范围                         |
|------------|------------------------------|
| admin      | 所有需求                         |
| sales      | 公开需求 + 自己创建的保密需求             |
| researcher | 公开需求 + 自己创建的保密需求 + 自己承接的保密需求 |

### 2.4 附件与下载追踪

- 研究员在"标记完成"时上传附件 (单文件)
- 任何有权限的用户下载附件时, 自动记录: `request_id`, `user_id`, `org_name`, `timestamp`
- 管理端可查看: Top10 下载排行、下载人数统计、近期下载记录

### 2.5 需求动态 (公开浏览)

- 销售端和研究端均可浏览 **已完成的公开需求**
- 支持多维筛选: 需求类型 / 研究范畴 / 客户类型 / 时间区间
- 支持附件快捷下载
- 支持导出 Excel

### 2.6 管理端统计分析

**工作量看板**:

- 时间维度选择: 今日 / 本周 / 本月 / 当季 / 今年
- 总览卡片: 需求总数 / 待处理 / 处理中 / 已完成 / 总工时
- 研究员排行: 按完成数排序, 展示工时, 可下钻查看明细

**多维分析** (6 个视角):

| 视角     | 维度                | 展示                |
|--------|-------------------|-------------------|
| 统计看板   | 需求类型 + 客户类型 + 研究员 | 饼图 + 柱状图          |
| 研究员视角  | 研究员 × 时间维度        | 多列表格 (今日/周/月/季/年) |
| 需求类型视角 | 需求类型 × 时间维度       | 同上                |
| 客户视角   | 机构 × 工时           | 表格                |
| 销售视角   | 销售 × 时间维度         | 多列表格              |
| 下载统计   | Top10 + 人数 + 明细   | 排行 + 表格           |

**数据导出**:

- 多条件筛选: 时间 / 类型 / 范畴 / 人员 / 机构 / 状态
- 预览前 20 条
- 导出 Excel (含下载次数列)

### 2.7 系统管理 (admin)

| 模块   | 功能                           |
|------|------------------------------|
| 用户管理 | CRUD + 密码重置 + 角色/团队分配        |
| 需求管理 | 编辑任意字段 / 重新分配研究员 / 切换保密 / 删除 |
| 机构管理 | CRUD 机构 (name + org_type)    |
| 团队配置 | 创建团队 / 分配机构到团队 / 分配销售到团队     |

### 2.8 团队与机构权限

```
团队 (teams)
  └── 机构映射 (team_org_mapping)
        └── 机构 (organizations)

销售 (users.team_id)
  → 只能看到/选择所属团队关联的机构
```

## 3. 明确排除 (v3.0 不做)

- ❌ 微信小程序 / uni-app 移动端 (后续基于同一套 API 扩展)
- ❌ 邮件/消息通知
- ❌ 工作流引擎 / 审批流
- ❌ 多租户 / SaaS
- ❌ PostgreSQL 迁移 (保留 SQLite, 架构预留切换能力)
- ❌ AI 辅助功能

---

## 4. 相比现有系统的改进点

| 维度   | 现状 (Streamlit)     | 重构后 (FastAPI + React)  |
|------|--------------------|------------------------|
| 交互   | 全页刷新, 表单受限         | SPA 局部更新, Modal/Drawer |
| 并发   | 每用户独立进程            | API 无状态, 轻量高效          |
| 移动端  | 勉强可用               | 响应式布局, 后续可出 App        |
| 权限路由 | session_state 手动控制 | 路由级 + 按钮级权限            |
| 安全   | SHA256 明文哈希        | bcrypt + JWT           |
| API  | 无, 紧耦合             | RESTful, 可对接外部系统       |
| 部署   | Python 进程          | 前端 CDN + 后端独立扩展        |

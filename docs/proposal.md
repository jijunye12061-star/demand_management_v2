# proposal.md — 系统提案

---

## 1. 项目背景

现有「需求管理系统」基于 Streamlit 构建，已稳定运行 4 个月，支撑销售→研究员→管理员的需求流转。  
随着功能增长，Streamlit 在交互体验、移动端适配、并发性能方面已接近瓶颈。  
本次重构目标：**以前后端分离架构重写，100% 覆盖现有功能，同时解决已知技术债务**。

---

## 2. 核心功能边界

### 2.1 用户与认证

| 功能   | 说明                                 | 角色    |
|------|------------------------------------|-------|
| 登录   | JWT 认证, 替代 Streamlit session_state | 全部    |
| 修改密码 | 用户自行修改                             | 全部    |
| 用户管理 | 创建/删除/重置密码/分配角色与团队                 | admin |

**角色模型**:

- `sales` — 销售: 提交需求, 查看进度
- `researcher` — 研究员: 处理任务, 上传结果, 也可代提需求
- `admin` — 管理员: 全局可见, 统计分析, 系统管理

> **Admin 兼容规则**: admin 同时具备研究员和销售身份。所有选择研究员/销售的下拉列表均包含 admin 用户。  
> 即 `GET /users/researchers` 返回 `role IN ('researcher', 'admin')`，`GET /users/sales` 返回
`role IN ('sales', 'admin')`。  
> admin 被选为销售时, 机构列表显示全部机构 (不受团队限制)。

### 2.2 需求流转 (核心工作流)

```
销售提交需求 ──────→ 待处理 (pending)
                        │
         ┌──────────────┼──────────────┐
         ↓              ↓              ↓
    研究员接受      研究员退回        管理员重新分配
    → 处理中      → 已退回           → 更换研究员
   (in_progress)  (withdrawn)
         │              │
         ↓              ├─→ 销售修改后重新提交 → pending
         │              └─→ 销售取消需求 → canceled
    上传附件 + 填写说明 + 记录工时
         │
         ↓
    标记完成 (completed)
```

**状态枚举**: `pending` | `in_progress` | `completed` | `withdrawn` | `canceled`

**研究员退回流程**:

1. 研究员对 `pending` 状态的需求执行退回操作, 必须填写**退回原因**
2. 需求状态变为 `withdrawn`, 保留 `researcher_id` (记录谁退回的, 用于审计)
3. 销售端看到「已退回」状态 + 退回原因, 可选择:
    - **修改并重新提交**: 编辑需求详情 + 重新选择研究员 → 状态回到 `pending`
    - **取消需求**: 状态变为 `canceled` (软删除)

**销售编辑权限**: 销售可编辑自己创建的 `pending` 或 `withdrawn` 状态的需求 (状态未进入 `in_progress` / `completed`)。

**需求提交字段**:

| 字段              | 必填 | 说明                                 |
|-----------------|----|------------------------------------|
| title           | ✅  | 事项名称                               |
| description     | —  | 内容描述                               |
| request_type    | ✅  | 枚举: 基金筛选/传统报告定制/量化策略定制/系统定制/综合暂时兜底 |
| research_scope  | —  | 枚举: 纯债/固收+/权益/量化/资产配置/其他           |
| org_name        | ✅  | 从用户所属团队的机构列表中选择                    |
| org_type        | 自动 | 随机构带入                              |
| department      | 条件 | 银行/券商/保险时必选子部门                     |
| researcher_id   | ✅  | 指定研究员 (含 admin)                    |
| is_confidential | —  | 默认公开                               |
| created_at      | —  | 默认当前时间, 支持回溯选择                     |

**研究员提交需求**: 额外需要选择 `sales_id` (代哪个销售提, 含 admin)

### 2.3 保密机制

| 角色         | 可见范围                         |
|------------|------------------------------|
| admin      | 所有需求                         |
| sales      | 公开需求 + 自己创建的保密需求             |
| researcher | 公开需求 + 自己创建的保密需求 + 自己承接的保密需求 |

### 2.4 附件与下载追踪

- 研究员在"标记完成"时上传附件 (单文件)
- 文件存储路径: `uploads/{request_id}/filename`, 按需求 ID 归档, 为后续多文件扩展预留结构
- 任何有权限的用户下载附件时, 自动记录下载日志
- **销售下载时**: 弹窗选择机构 (从自己团队的机构列表中选), 记录该机构用于后续跟踪 (场景: 机构B对机构A的需求成果感兴趣)
- **研究员/admin 下载时**: `org_name` 记录为 null
- 管理端可查看: Top10 下载排行、下载人数统计、近期下载记录

### 2.5 需求动态 (公开浏览)

- 销售端和研究端均可浏览 **已完成的公开需求**
- 支持多维筛选: 需求类型 / 研究范畴 / 客户类型 / 时间区间
- 支持附件快捷下载

**展示字段** (仅以下字段对外可见):

- ✅ title, description, request_type, research_scope, org_type, researcher_name, completed_at, attachment (可下载)

**隐藏字段** (后端 scope=feed 时不返回):

- ❌ org_name, department, work_hours, sales_name, is_confidential

**导出 Excel**:

- 所有角色均可导出需求动态, 但 **仅包含 feed 可见的公开数据和展示字段**
- admin 通过管理端「数据导出」页导出完整数据 (全字段)

### 2.6 管理端统计

- 工作量看板: 总数/待处理/处理中/已完成/总工时 + 研究员排行
- 多维分析: 研究员/需求类型/客户/销售/下载 6 个维度 + 图表
- 数据导出: 多条件筛选 + 预览 + 全字段 Excel 下载 (admin only)
- 系统管理: 用户/需求/机构/团队 CRUD

## 3. 明确排除 (本期不做)

| 功能         | 原因                 |
|------------|--------------------|
| 邮箱/手机找回密码  | 用户体量小, admin 重置够用  |
| 多文件上传      | 本期维持单文件, 目录结构已预留扩展 |
| 消息通知       | 暂无邮件/IM 集成         |
| 审批流        | 当前需求流转足够, 无多级审批需求  |
| 国际化 (i18n) | 纯中文系统              |
| 移动端原生 App  | Web 响应式优先          |
